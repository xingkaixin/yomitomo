00 · Overview

聚焦共读全局流程

聚焦共读不是普通的"让一个助手全文批注"。它先生成章节级计划，再按计划拆分给每位助手独立执行，同一划线的 AI 输出合并为已有批注下的新想法。

核心约定：计划保存在 ArticleRecord.focusCoReadingPlan，批注和想法保存在 ArticleRecord.annotations。计划阶段只做分配，执行阶段才生成真实 Annotation。
01 · Data Model

核心数据模型

聚焦共读引入三个主要类型：计划级的 FocusCoReadingPlan 和 SectionPlan，执行级的 AgentReadingPlanItem。

FocusCoReadingPlan

id / articleId计划和文章唯一标识
selectedAgentIds参与规划的助手池（不代表每章都执行）
sections[]章节编排列表 SectionPlan
readingMemory执行产生的压缩阅读记忆

FocusCoReadingSectionPlan

sectionId / Title章节标识和标题
sectionStart / End在 article.text 中的字符范围
summary / tagAI 生成的摘要和标签
targetDensity批注密度建议 low/medium/high
agentIds[]本章节分配的助手
messages[]读者留言（可指定助手）

AgentReadingPlanItem

sectionId / Title执行目标章节
sectionStart / End落锚允许的字符范围
sectionSummary / Tag传递给 AI 的章节提示
readingIntent阅读动作意图
targetDensity覆盖默认密度
messages[]只包含对当前助手生效的留言

FocusCoReadingMessage

content读者留言文本
agentId指定的目标助手（单个，可选）
agentIds[]指定的多个目标助手（可选）
agentNicknames[]被 @ 的助手昵称
02 · Reading Sections

阅读章节来源

聚焦共读基于 ReaderReadingSection[] 编排。网页和 EPUB 各自有不同的章节生成方式。

🌐 网页文章
buildReaderReadingSections(articleRef, tocItems, title)

从文章 DOM 和 TOC 推导章节。有正文前引文时生成 intro 章节；无可用 TOC 时退化为单个 body 章节。
📖 EPUB 电子书
ebookReaderReadingSections(article, ebookText)

使用 EPUB index 和章节信息生成。sectionStart/End 对应全书聚合文本偏移。
接入要求：新阅读器必须提供稳定的章节 id 和字符范围。自动分析、手工编排、执行落锚和无剧透上下文都依赖这些范围。
03 · Auto Analysis

选择助手与自动分析

用户选择助手后触发 AI 自动分析文章，生成章节级摘要、标签和助手分配。

🔍

planCoReading() 自动分析流程

focus-co-reading:route
1
用户点击"添加助手"，id 加入 selectedAgentIds
2
点击"开始分析文章"触发 planCoReading()
3
调用宿主 onPlanFocusCoReading(selectedAgentIds) → preload → focus-co-reading:route
4
main 读取 store，组装 payload（selectedAgentIds, sections, chapterSummaries, article）
5
AI 返回 FocusCoReadingRouteResult：每章的 summary, tag, agentIds, targetDensity
6
合并为新 FocusCoReadingPlan：继承旧 messages/readingMemory/createdAt，保存到文章
合并规则：route 返回的 agentIds 只保留用户本轮选择的助手。旧计划的 messages 按 section 继承。没有助手、留言、summary、tag 的 section 不保存。
04 · Manual Arrangement

手工编排助手

用户可以在自动分析前后手工调整章节的助手分配和留言。每次调整都即时保存。

🎛️ 编排操作与数据流
添加全局助手
addPlanAgent(id)
加入 selectedAgentIds 池
移除全局助手
removePlanAgent(id)
从所有章节移除 + 过滤留言
章节添加/移除助手
add/removeSectionAgent()
移除时过滤只发给该助手的留言
清空所有章节
clearSectionAssignments()
清空所有章节助手和留言
每次调整 → saveSections() → onSaveFocusCoReadingPlan(plan) → onUpdateArticle() → article:save
05 · Section Messages

添加章节留言

留言是给章节或章节内特定助手的阅读指引。支持 @助手 指定目标，不指定时对所有助手生效。

💬

留言创建与过滤

1
用户展开章节 → 输入留言（可用 @助手昵称 指定目标）
2
focusMessageFromDraft() 从留言中提取被 @ 的章节助手
3
匹配到 → 写入 agentIds/agentUsernames/agentNicknames；未匹配 → 全局留言，对所有助手生效
4
助手被移除章节时，filterFocusMessagesForAgents() 过滤失效留言
过滤规则：留言没有指定目标助手时，只要章节还有任意助手就保留。指定了多个助手时，只保留仍在章节中的目标。
06 · Plan → Execution

计划转执行

开始共读时，按助手拆分 SectionPlan 为各自的 AgentReadingPlanItem[]，并对留言做二次路由。

🚀

startReadingPlan() 拆分流程

1
遍历所有可用助手，从 sections 中筛选包含该助手 id 的章节
2
focusSectionToReadingPlanItem() 转成 AgentReadingPlanItem，只保留对当前助手生效的 messages
3
routeFocusReadingPlanMessages() 二次路由留言：@A 解释概念 → A 看到"解释概念"，@B 挑战前提 → B 看到"挑战前提"
4
调用 onStartAgentPlan(agent, readingPlan) → requestAgentAnnotations(agent, { readingPlan })
留言二次路由：对含 @ 的留言调用 agent:mention-route（只允许 create_thought），把 content 替换成该助手的具体 instruction。路由失败时退回 agentInstructionFromNote() 去掉 @ 称呼后的文本。
07 · Execution

执行共读：生成批注

通过 agent:annotate:stream 流式生成批注。每条 AI 输出都经过章节范围验证后才进入保存链路。

🤖

执行共读主流程

agent:annotate:stream
1
promptArticle() + buildAgentAnnotationRequestInput() 构造 payload
2
routeFocusReadingPlanMessages() 二次路由留言
3
runSourceAgentAnnotationRequest() → preload → main runAgentAnnotateStream()
4
收到 item → constrainAgentPlanAnnotation() 验证落在 reading plan 章节范围内
5
验证通过 → 播放/保存链路 → appendAgentAnnotationToArticle()
6
EPUB 多 segment 时，每个 segment 后合并 readingMemory，最终 saveFocusCoReadingReadingMemory()
🌐 网页执行差异
startAgentAnnotationPlayback() 启动虚拟阅读动画。当前文章走 enqueueAgentAnnotation() 边播放边保存；后台 article 直接写入。
📖 EPUB 执行差异
startEbookPlayback() 启动 dock 播放。通过 Foliate 页面定位。批注不在当前可见范围时尝试跳转或直接保存。
08 · Assistant Theater

助手演出链路

执行共读时有两条并行链路：AI 生成和保存链路负责产出 Annotation，前端演出链路负责展示助手正在读、移动到落点和完成收尾。演出状态不写入 ArticleRecord。

dock 栏状态机

activateactivateAgentDock(agent) 进入 active
donemarkAgentDockDone(agentId) 单个助手完成
completecompleteAgentDock(celebrate) 全部结束后收尾
burst成功时递增 completionBurstKey，显示完成 burst
clearclearAgentDock() 取消、卸载或清理时清空

虚拟鼠标状态机

start根据 reading plan、target anchor 或阅读容器生成坐标
update写入内存 map，ReaderAppView 渲染 VirtualCursor
finish停止 timer，更新离场 label，延迟移除 cursor
offscreen目标在视口外时显示上方/下方提示
cleanup清理 interval/timeout、cursor、dock、失败标记

agentTheaterBoxes

定位从 DOM Range、Foliate range 或 PDF anchor geometry 生成
用途播放过程中展示临时高亮
保存不保存；最终高亮来自 Annotation.anchor
fallback不可见或无法定位时直接保存，或展示 dock-only / 边缘提示
🌐 网页文章
startVirtualReading(agent, readingPlan, mode) 创建 VirtualReadingSession，根据 section offset 持续更新 cursor。careful 和 target 模式显示 dock；播放队列空闲后 finishVirtualReadingIfIdle() 触发离场。
📖 EPUB
startEbookPlayback() 启动 dock；目标选区模式才启动虚拟阅读。Foliate anchor 可定位时跟随选区，不可定位时 fallback 到阅读容器顶部区域。cleanupAgentTheater() 清理 timers、cursor、dock 和临时高亮。
📄 PDFium
PDFium 当前支持单次助手批注的 dock、虚拟鼠标和临时高亮演出，但 onStartAgentReadingPlan 仍是空实现，尚未接通聚焦共读入口。定位优先使用 PDF text anchor 和页面 metrics，失败时 fallback 到画布顶部。
🎬

播放与保存边界

1
AI 输出通过章节范围验证后进入播放队列；当前文章播放，后台文章直接保存。
2
播放层尝试定位锚点，移动虚拟鼠标并展示 agentTheaterBoxes；不可见内容使用边缘提示、容器 fallback 或直接保存。
3
最终保存仍调用 mergeAgentAnnotationAsThought() / appendAgentAnnotationToArticle()，不依赖演出状态。
09 · Merge Logic

相同划线合并为新想法

同一段划线文本上的多个 AI 输出不会创建重复批注卡片，而是合并为已有批注下的新顶层想法。

mergeAgentAnnotationAsThought() 合并决策

✅ 找到相同 exact key

把 AI annotation 的 comments 追加到已有批注的 comments 中。replyTo 置空，成为新顶层想法。已存在相同 comment id 时跳过。返回 activeId = existing.id。

OR

🆕 未找到相同 exact key

新增一条 Annotation，创建新的批注卡片。

合并边界：基于 annotationExactKey() 做文本匹配，不依赖 AI annotation id。只合并 comment，不合并 annotationType/moveType/whyHere 到已有批注顶层。不同 exact 但语义相近的文本不会合并。
10 · Prompt Design

Prompt 设计

计划阶段和执行阶段使用不同的 prompt 策略。

网页文章 Route

buildFocusCoReadingRoutePrompt()
  • 文章标题、URL
  • 助手列表（id, username, nickname, roleCard）
  • 章节清单 + 压缩文本（≤1200 原样，>1200 前 900 + 后 240）
输出
{ sections: [{ sectionId, summary, tag, agentIds }] }

EPUB Route

buildEpubFocusCoReadingRoutePrompt()
  • 书籍元数据（标题、作者、语言、章节数）
  • 助手列表 + 完整角色卡
  • 章节 descriptors（id, title, length, segmentCount, preview）
  • 压缩 chapter route context
额外字段
targetDensity, needsFurtherPlanning

非 segment 执行

buildAgentAnnotateStreamPrompt() reading plan 分支
  • 文章标题 + URL + 预算裁剪文本
  • readingPlanPrompt()：sectionText, summary, tag, readerMessages
  • 无剧透范围 + 批注密度指令
  • exact 必须来自 sectionText 连续原文
约束
只能在编排的 sectionText 内落锚

EPUB segment 执行

buildAgentSegmentAnnotateStreamPrompt()
  • book / chapter / currentSegment 信息
  • routeInstruction（summary, tag, intent, messages）
  • allowedAnchorRange（coreParagraphIds）
  • packed blocks：segment, evidence, memory, trace, dedup
约束
只能在 currentSegment coreParagraphIds 内落锚

阅读记忆更新

generateSegmentReadingMemoryUpdate()
  • 当前 chapter / segment 范围和原文
  • previousSummary + previousSegmentTrace
  • chapterTrace + routeInstruction
  • 当前助手身份 + 本 segment 已接受 annotations
输出
{ segmentSummary, segmentTrace, chapterTrace }
11 · IPC & Storage

存储与 API

聚焦共读涉及的 IPC channel 和持久化位置。

Channel类型用途
focus-co-reading:route计划自动分析文章并生成章节级分配
agent:annotate:stream执行执行共读并流式返回批注
agent:mention-route路由章节留言二次路由（@ 助手拆分）
article:save保存保存计划、readingMemory、批注和评论
持久化位置：FocusCoReadingPlan 存在 articles.focus_co_reading_plan JSON 字段中。批注仍在 annotations 和 comments 表。readingMemory 在计划 JSON 内。
12 · Integration Checklist

新阅读器接入清单

接入聚焦共读功能时需要满足的全部要求。

1
提供稳定的 ReaderReadingSection[]，包含 id、标题、开始/结束文本 offset
2
把 focusCoReadingPlan 从 ArticleRecord 传给 ReaderAppView
3
实现 onPlanFocusCoReading()，调用 focus-co-reading:route
4
实现 onSaveFocusCoReadingPlan()，通过 onUpdateArticle() 写回
5
实现 onStartAgentReadingPlan()，调用 requestAgentAnnotations()
6
确保 promptArticle() 的 text 与 section offset 使用同一份文本
7
AI 输出经过 constrainAgentPlanAnnotation() 验证章节范围
8
保存时经过 mergeAgentAnnotationAsThought() 确保相同划线合并
9
长文/电子书提供 index、segment、paragraph 信息，否则无法使用 segment 上下文和 readingMemory
10
支持共读演出时，把 Annotation.anchor 或目标选区映射为 viewport 坐标
11
锚点不可见或无法定位时提供稳定 fallback，例如容器顶部、上下边缘提示或 dock-only 演出
12
播放队列开始、单个助手完成、全部助手完成和失败取消时调用对应 dock 状态方法
13
卸载、文章切换和播放取消时清理 interval/timeout、虚拟光标、agentTheaterBoxes、dock 状态和 active id 记录
14
不要把演出状态作为保存成功依据；最终事实仍是 ArticleRecord.annotations
