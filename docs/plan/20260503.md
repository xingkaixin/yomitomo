# Yomitomo 产品打磨与功能迭代计划

日期：2026-05-03

## 产品定位

Yomitomo 是一个伴读产品。核心体验发生在阅读现场：用户在文章中阅读、选择、批注、追问，多个助手围绕原文参与讨论。桌面端承担本地配置、长期记忆、知识产物和复盘入口；浏览器插件承担网页阅读、原文锚定、即时交互和低打扰陪读。

产品目标是让用户读得更顺、想得更深、留得下来。功能优先服务这条链路：阅读现场 -> 原文批注 -> 多轮讨论 -> 读后沉淀 -> 后续回访。

## 基本原则

1. 伴读优先  
   功能围绕阅读过程展开，避免把产品推进成泛聊天、泛收藏或泛知识库。

2. 原文锚定  
   批注、评论、AI 判断都应尽量回到原文片段。可信度来自“观点能回跳到来源”。

3. 低打扰主动性  
   AI 可以主动读、主动批注、主动提示，但主动性需要可控，用户应能调节批注密度和助手参与节奏。

4. 桌面端作为记忆中心  
   插件负责现场交互，桌面端负责保存文章、批注、讨论、统计和知识产物。

5. 小步闭环  
   每次迭代优先打通一个完整闭环，减少一次性大改和跨模块扩散。

## 近期规划

### R1：文章记录回流桌面端

目标：让插件端产生的文章和批注回到桌面端，形成阅读库雏形。

功能范围：

- 插件在保存批注时，将文章记录同步给桌面本地服务。
- 桌面端 store 保存文章记录。
- 桌面端新增“阅读库”入口，展示文章标题、来源、更新时间、批注数、评论数。
- 阅读库展示基础汇总：文章数、批注数、讨论数。

验收标准：

- 用户在插件阅读器中创建用户批注或 AI 批注后，桌面端刷新可以看到对应文章。
- 同一文章多次更新会覆盖同一条记录。
- 桌面端关闭重开后文章记录仍然存在。

### R2：读后卡片

目标：把一次伴读过程转化为可复用知识产物。

功能范围：

- 在桌面端为单篇文章生成读后卡片。
- 卡片包含：核心判断、关键原文、用户批注、AI 补充、后续问题。
- 当前版本基于已同步的文章、批注和评论做规则整理。
- 卡片以 Markdown 呈现，支持一键复制。

验收标准：

- 一篇带批注文章可以生成一张可复制的读后卡片。
- 用户可以区分原文内容、自己的想法和助手观点。

后续增强：

- 接入助手生成更自然的核心判断和问题整理。
- 卡片保留可回跳的原文锚点。

### R3：基础统计

目标：让用户看到阅读投入和产出，形成轻量激励。

功能范围：

- 今日/本周阅读文章数。
- 批注数、评论数、AI 参与次数。
- 累计阅读文章数、批注数、评论数、AI 参与次数。
- 阅读器停留时间后续由插件端记录后同步。

验收标准：

- 桌面端可以看到当天和本周的基础统计。
- 桌面端可以看到累计统计。
- 统计来源于本地记录。
- 当前版本不展示虚假的阅读分钟；阅读时长等插件端停留时间数据落库后再展示。

### R4：批注类型与批注密度

目标：提升 AI 自动批注的可读性和节奏感。

功能范围：

- AI 批注增加类型：关键判断、前提漏洞、概念解释、延伸问题、金句。
- 助手增加批注密度：克制、标准、积极。
- 插件侧边栏支持按类型扫读批注。

验收标准：

- AI 自动批注带有稳定类型。
- 不同密度下批注数量有明显差异。

## 中期规划

### M1：知识笔记

目标：桌面端从阅读库升级为知识产物中心。

功能范围：

- 从文章记录生成 Markdown 笔记。
- 笔记结构包含文章信息、核心观点、批注摘要、讨论摘录、待追问问题。
- 支持复制和导出。

### M2：跨文章关联

目标：让助手具备记忆连续性。

功能范围：

- 根据标题、正文、批注和标签建立文章间关联。
- 阅读新文章时提示历史相关文章和历史问题。
- 桌面端支持按主题查看相关文章。

### M3：复盘页

目标：形成周/月级阅读回顾。

功能范围：

- 近期读过什么。
- 哪些主题反复出现。
- 哪些问题仍未解决。
- 哪些批注值得整理成知识卡片。

### M4：导出与迁移

目标：降低用户迁移顾虑，增强知识资产信任。

功能范围：

- Markdown 导出。
- Obsidian 友好格式。
- 后续评估 Notion、PDF、JSON。

## 远期规划

### L1：桌面端文档阅读器

目标：把伴读能力从网页扩展到本地文档。

内容入口：

- PDF
- Word
- ePub
- 本地 Markdown

核心要求：

- 复用原文锚定、批注、讨论、多助手参与。
- 本地文件记录进入同一阅读库。

### L2：长期记忆

目标：让助手逐渐理解用户阅读偏好和知识背景。

能力方向：

- 记住用户常用追问方式。
- 记住用户关心的主题。
- 记住用户对某些概念的已有理解。
- 助手回复风格保持稳定。

### L3：个人知识图谱

目标：从文章列表升级为可探索的思想索引。

实体类型：

- 文章
- 批注
- 问题
- 概念
- 作者
- 主题

### L4：本地模型与隐私模式

目标：支持隐私敏感场景。

能力方向：

- Ollama 或兼容 OpenAI API 的本地模型。
- 敏感文章仅本地处理。
- 用户明确选择云端处理范围。

## 当前第一次迭代

本次开始 R1：文章记录回流桌面端。

开发任务：

1. 扩展 `DesktopStore`，增加 `articles`。
2. 桌面端 store 增加文章保存逻辑。
3. WebSocket 协议增加 `article:save`。
4. 插件端保存批注后发送文章记录。
5. 桌面端新增“阅读库”视图。
6. 跑通 build/test。

本次暂缓：

- 文章全文入库。
- 搜索、标签、导出。
- 阅读时长统计。
- 读后卡片生成。

这些能力依赖桌面端文章记录稳定存在，后续迭代基于 R1 增量推进。

## R1.0.1：日志查看与诊断入口

目标：把日志能力从侧边栏临时路径展示，收敛到“关于”设置页，形成可用的本地诊断入口。

功能范围：

- “关于”页新增“日志”模块。
- 显示日志路径，并支持复制路径。
- 打开页面时加载当前日志内容。
- 日志内容不实时读取，用户通过“刷新”手动读取新内容。
- 支持搜索日志内容。
- 支持按日志级别筛选：全部、Info、Error。
- 支持清理日志文件。

后续记录：

- 未来可以增加“上报日志”能力，用于用户主动提交诊断信息。
- 上报前必须提供用户确认、可预览内容和敏感信息处理策略。

## R1.1：桌面端存储切换到 SQLite + Drizzle

开发阶段直接使用新的 SQLite 存储，不迁移旧 JSON 数据。第一版发布后，所有 schema 变更必须通过版本化迁移完成。

目标：

- 桌面端持久化从 JSON 文件切换为 SQLite。
- 使用 Drizzle 定义 schema 和读写模型。
- 保留应用内 migration runner，启动时自动执行未应用迁移。
- `better-sqlite3` 作为 SQLite driver，开发和构建前通过 `electron-rebuild` 重建 Electron ABI。
- 为后续全文检索、自然语言检索、统计和读后卡片提供稳定数据模型。

当前 schema：

- `user_profiles`：用户身份和批注颜色。
- `providers`：LLM 供应商配置。
- `agents`：助手身份、头像、个性和 provider 关联。
- `articles`：文章元信息和内容 hash。
- `annotations`：原文锚点、作者和批注身份。
- `comments`：批注 thread 内的用户和助手评论。

迁移约定：

- 首版前可以直接调整 `0001_initial`。
- 首版发布后，新增 schema 变更通过追加 migration id 完成。
- 迁移必须只做确定性结构变更和必要数据转换。
- 业务读写通过 store API 访问数据库，UI 和 WebSocket 层不直接读写 Drizzle。
