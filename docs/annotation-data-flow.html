01 · Data Model

核心模型

批注数据挂在 ArticleRecord.annotations 上。每条划线是一个 Annotation，每条想法或回复是一个 Comment。

TextAnchor

exact用户选中的原文文本
prefix / suffix选区前后文，用于模糊恢复
start / end在全文 textContent 中的字符偏移
paragraphId所在段落 ID（可选）
chapterId所在章节 ID，电子书推荐
textStartInBook全书偏移，EPUB 推荐
quoteHash选区文本哈希，加速匹配

Annotation

id唯一标识
anchorTextAnchor，划线定位唯一事实来源
author"user" | "ai"
comments[]想法列表和回复树
annotationTypeAI 推断的批注类型
readingIntent阅读动作意图
color划线颜色

Comment

id唯一标识
author"user" | "ai"
content想法或回复的文本内容
replyTo为空 = 顶层想法，有值 = 回复
reviewLabel审阅标签（审阅回复专用）
pendingtrue = 流式占位，不持久化

05 · Persistence

保存与持久化

批注保存当前是"文章级替换"：删除该文章所有旧 annotations，按当前状态重建。并发写入应通过 onUpdateArticle 队列合并。

Renderer saveAnnotations(next) → onSaveArticle(nextArticle)
↓
IPC Preload window.yomitomoDesktop.saveArticle(article) → article:save
↓
Main Store saveArticle(input) → writeArticleRows() + upsert annotations + comments
↓
SQLite articles · annotations · comments （复杂字段 JSON 存储）
三个保存语义： saveAnnotations() 排序 + 落库 · applyAnnotations() 仅更新本地状态（流式占位）· replaceAnnotations() 外部更新替换本地
06 · IPC Contract

Desktop IPC 清单

所有 renderer ↔ main 进程通信 channel，分为文章保存和 AI 调用两大类。

Channel类型用途
article:get保存读取完整文章和批注
article:save保存保存文章、批注、评论（文章级替换）
article:reading-progress保存保存阅读进度
annotation:metadataAI推断批注类型与阅读动作
agent:mention-routeAI@助手 路由：是否保存想法 + 助手动作
agent:commentAI非流式助手回复
agent:comment:streamAI流式助手回复
agent:annotateAI非流式助手创建批注/想法
agent:annotate:streamAI流式助手创建批注/想法
agent:reviewAI审阅助手批量审阅顶层想法

08 · AI Thought Pipeline

AI 新想法链路

助手围绕选区或主动在文章中创建批注和想法的流程。模型输出的 exact 必须在原文中匹配才会保存。

💡

agent:annotate:stream

app-source-agent-request.ts
1

2
调用 requestAgentAnnotationsStream(payload, onEvent)
3
main 执行 runAgentAnnotateStream()，AI 输出 JSON/NDJSON suggestion
4

6
target 模式下 targetAnchorSuggestion() 强制覆盖 exact 为用户选区，防漂移

10 · Prompt Assembly

Prompt 拼接规则

不同场景下 system prompt 和 user prompt 的拼接内容和策略。

批注元数据推断

buildAnnotationMetadataPrompt()
  • 文章标题 + URL
  • 用户选区 anchor.exact
  • 用户批注 note
  • 批注类型候选 + 阅读动作候选
输出
{ annotationType, readingIntent }

@助手语义路由

buildAgentMentionInstructionPrompt()
  • 文章标题 + URL + 目标选区/章节
  • 用户文本 + @助手列表
  • 可选阅读动作 + 允许动作
输出
{ createUserThought, directives[] }

助手回复

buildAgentPrompt()
  • 角色卡 + 阅读动作 system prompt
  • EPUB: thread-first 上下文 + lexical 召回
  • 普通: 全文预算裁剪 + 高亮 + 讨论
  • 无剧透范围 + 参与者 + 身份规则
路径选择
thread-first (EPUB) / budget-text (web)

助手新想法

buildAgentAnnotatePrompt()
  • 角色卡 + target/主动/reading plan 三模式
  • target: 选区 + 局部窗口 + 章节记忆
  • 主动: 全文裁剪 + 批注密度说明
  • exact 必须从 sectionText/选区匹配
输出
JSON / NDJSON suggestion[]

审阅

buildAgentReviewPrompt()
  • 角色卡 + 审阅标签说明
  • 全文预算裁剪 + 无剧透范围
  • 用户高亮 + 待审阅想法 JSON
  • 已审阅 thought id 排除
输出
[{ thoughtId, label, content }]
11 · Integration Checklist

新阅读器接入清单

接入新阅读器实现时需要满足的全部要求。

1
提供文章纯文本，用于 promptArticle()、锚点定位和 AI exact 匹配
2
选区生成 TextAnchor：普通文档用 createTextAnchor()，电子书优先建 index 用 EPUB 锚点
3
实现高亮 boxes：根据 annotationsRef 和 viewport 计算每条批注的 DOM/画布矩形
4
复用 useSourceAnnotations()，不要重新实现评论追加、删除、保存和 stale 防护
5
接入 AnnotationCard / ReaderAppView 时，把 onAddComment、onDelete、onDeleteComment 传回 hook
6
保存统一走 onSaveArticle() 或 onUpdateArticle()，最终落到 article:save
7
AI 回复复用 runSourceAgentCommentRequest()
8
AI 创建想法复用 buildAgentAnnotationRequestInput() 和 runSourceAgentAnnotationRequest()
9
@助手路由调用 planSelectionMentionRoute()，不要硬编码语义
10
电子书/长文传入 ebookIndex、阅读进度和章节 id，否则 AI 退化为全文预算裁剪
